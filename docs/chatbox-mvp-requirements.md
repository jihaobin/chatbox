# Chatbox MVP 需求梳理（仅支持 OpenAI）

## 1. 项目目标

- 在最短周期内交付一个可用的 Chatbox MVP。
- MVP 阶段仅支持 OpenAI 系列模型调用，确保核心对话链路稳定可用。
- 面向零技术门槛用户，前端无需配置 API Key 即可直接发起对话。
- 架构需预留模型提供方扩展能力，便于后续接入其他模型。

## 2. 范围定义

### 2.1 MVP 包含（In Scope）

- 基础聊天能力：用户输入问题、模型返回回答。
- 会话管理：支持新建会话、查看历史消息（单用户场景）。
- OpenAI 模型配置：由系统在服务端统一配置 API Key 与默认模型参数（如 temperature、max tokens）。
- 错误处理：对鉴权失败、限流、网络异常、超时等场景给出可读提示。
- 基础日志：记录关键请求链路（请求时间、模型、状态码、耗时）。

### 2.2 MVP 不包含（Out of Scope）

- 非 OpenAI 提供方模型接入（如 Anthropic、Gemini、本地模型）。
- 多用户权限体系与团队协作能力。
- 高级工作流（Agent 编排、工具调用、插件市场）。
- 复杂计费系统与精细化配额管理。

## 3. 角色与核心用户故事

- 作为用户，我可以输入问题并收到模型回复。
- 作为用户，我可以看到当前会话的历史消息。
- 作为用户，我无需配置任何 API Key，打开应用即可直接对话。
- 作为用户，我在调用失败时能看到明确的错误原因与建议操作。

## 4. 功能需求清单

### 4.1 聊天界面

- 消息列表：按时间顺序展示用户消息与助手消息。
- 输入框：支持发送文本消息，支持发送中禁用状态。
- 发送状态反馈：展示加载中、成功、失败状态。

### 4.2 会话管理

- 新建会话。
- 切换会话并加载历史消息。
- 本地持久化最近会话（MVP 可先本地存储）。

### 4.3 OpenAI 配置与调用

- 配置项（服务端）：`OPENAI_API_KEY`、默认 `model`、`temperature`、`max_tokens`。
- 调用协议：封装统一的 OpenAI Chat Completions/Responses 调用层（二选一，项目内保持一致）。
- 安全要求：API Key 仅保存在服务端，不下发到前端，且不在日志中明文输出。
- 前端要求：不提供 API Key 输入入口。

### 4.4 错误与可观测性

- 错误分类：认证失败、参数错误、限流、网络异常、服务端错误。
- 前端提示：错误消息可读且可操作（如“服务暂不可用，请稍后重试”）。
- 基础日志：记录错误类型与上下文（脱敏）。

## 5. 非功能需求

- 响应性能：正常网络下首字返回时间可感知地快速（优先支持流式返回，若首版不做则需明确）。
- 稳定性：关键路径出现异常时不崩溃，用户可重试。
- 可维护性：模型调用层与业务层解耦，便于未来扩展模型提供方。

## 6. 技术与架构约束

- 仅接入 OpenAI 系列模型。
- 预留 Provider 抽象层，例如：`ModelProvider` 接口 + `OpenAIProvider` 实现。
- 所有模型调用经统一网关/服务模块，避免在 UI 层分散调用。
- 前端不得直接调用 OpenAI 接口，必须通过后端代理层转发请求。

## 7. 里程碑建议（MVP）

- 里程碑 1：完成聊天 UI + 单次消息调用。
- 里程碑 2：完成会话管理 + 本地持久化。
- 里程碑 3：完成服务端 OpenAI 配置能力 + 错误处理。
- 里程碑 4：完成日志与基础验收。

## 8. 验收标准（Definition of Done）

- 在前端零配置 API Key 的前提下，可完成至少一轮多轮对话。
- 会话可新建、切换、查看历史消息。
- 常见错误场景有明确提示，不出现未处理异常。
- 代码中模型提供方调用点集中，具备后续扩展空间。

## 9. 后续扩展方向（非本期交付）

- 增加多 Provider 支持（Anthropic/Gemini/本地模型）。
- 增加 Prompt 模板管理与会话导出。
- 增加用户体系与配额控制。
